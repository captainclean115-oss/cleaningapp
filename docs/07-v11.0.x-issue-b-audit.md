# v11.0.x Issue B — Comprehensive Hardcoded-Data Audit

**Date:** 2026-05-12
**Branch:** `fix/offices-per-tenant-and-audit` (commits c74f217 et al)
**Status:** Report only — no fixes applied beyond what's already in Issues A + C.

## Scope

After v11.0.3 (`_seedJobs` etc.) and v11.0.4 Issue A (office pins) + Issue C (chat seed) were fixed, this audit sweeps the rest of `index.html` for any remaining hardcoded tenant-specific operational data. Per Tom's expanded scope, employee-facing surfaces (portal, mobile, clock-in/out, job detail) are explicitly covered.

Categories scanned: large object/array literals, coordinate pairs, phone numbers, email addresses, per-tenant config, image URLs, third-party API endpoints, webhook URLs, business identifiers, customer-facing message templates, holiday calendars, tax configs, recurring schedule defaults, pricing rules.

## Findings — sorted by severity

### 🔴 CRITICAL — sensitive data exposed via JS bundle

#### Finding B-1: `CLIENT_KEYS` dump (line 34401)

**Variable:** `var CLIENT_KEYS = { 'ahern': {...}, 'schneider': {...}, ... }` — 5,147 bytes, single line. ~90 Manna Maids clients.

**Contents include:**
- Door codes: `"Door code: 4444"`, `"GARAGE CODE-1963"`, `"CODE 4249 TURN DEAD LOCK"`, `"Garage code 0607"`
- Property access: `"KEY UNDER MAT,PUT BACK"`, `"SIDE DOOR OPEN!"`, `"KEY  UNDER MAT."`
- Client first names + property keys (`{"key": "Door code: 4444", "fn": "Brady"}`)

**Severity:** Worse than Sullivan gate code 4422 (which Tom is rotating separately). Every browser that loads the production app receives ~90 client property access codes in plaintext.

**Should be:** Per-tenant — likely already covered by `client_keys` Supabase table (Migration 027 or earlier added it; need to verify). The hardcoded const is the legacy seed, identical anti-pattern to `_seedJobs`.

**Recommended fix:** delete the hardcoded const; verify the existing `client_keys` table has tenant-scoped RLS; if any of these codes are still operationally needed, they should be migrated into client_keys rows for Manna before the const is deleted, then the const goes to `{}`. **All gate/door codes in this dump should be considered compromised — recommend rotation at the property side for each.**

#### Finding B-2: `GEOTAB_USER` + `GEOTAB_PASS` (lines 23533–23534)

Already surfaced and acknowledged. Tom rotating password manually. Code-side migration to per-tenant `business_gps_integrations` table is queued as Item D for separate scoped work. **Until that ships, do NOT delete the const — the live Geotab path still references it.** Listing here for completeness.

#### Finding B-3: `GEOTAB_DB` (line 23532)

```js
const GEOTAB_DB = 'the_maids_of_the_south_shore';
```

Sibling to the Geotab credentials — Manna's specific Geotab database name. Should be per-tenant in the same `business_gps_integrations` table as the credentials. **Same Item-D scope, same deferral.**

### 🟠 HIGH — tenant data leaking via bundle but not credentials

#### Finding B-4: `WEATHER_LOCATION` (line 36675)

```js
const WEATHER_LOCATION = { lat: 42.0959, lon: -70.9495 };
```

Manna's Abington office coords baked in. Used to fetch OpenWeather forecast (line 36691). Every tenant's app fetches weather for Abington, MA regardless of where they actually operate.

**Should be:** Resolved from the tenant's primary office (`PentaOffices.listSync()[0].lat/lng`) or `businesses.address` jsonb. Falls back gracefully to no-weather if no office configured.

**Recommended fix:** ~10-line refactor — read coords lazily before the OpenWeather fetch.

#### Finding B-6: "Penta, Inc." literals in handbook + applicant flow (17 occurrences)

Conflates the SaaS platform name with the tenant identity. Categories:

- **Employee handbook welcome + at-will paragraphs** (8 occurrences across 4 language versions, lines 33063/33070/33168/33174/33243/33249/33318/33324):
  > "Welcome to Penta, Inc. We are proud to have you as part of our team..."
  > "Employment at Penta, Inc. is at-will..."

- **Job-application consent + success messages** (8 occurrences across 4 language versions, lines 30205/30209/30239/30243/30273/30277/30307/30311):
  > "I authorize Penta, Inc. to verify all information."
  > "Thank you for applying to Penta, Inc."

- **Static HTML in form footer** (lines 38735, 38752): same consent + success copy as above

- **Employee signature acknowledgment** (line 28375):
  > "the employee confirmed reading and agreeing to the Penta, Inc. Employee Handbook"

**Should be:** the *tenant's* legal name, not the platform's. Each operator's employees apply to and work for THAT operator's company, not Penta. `businesses.legal_name` column already exists; just unused.

**Recommended fix:** Add `{{BUSINESS_LEGAL_NAME}}` placeholder (same mechanism as the existing `{{BUSINESS_NAME_UPPER}}` and `{{BUSINESS_ADDR}}` from Issue A) and substitute at render time. ~12 occurrences in templates + 4-language application form translations object. Reads `businesses.legal_name` (with fallback to `name + ', Inc.'`).

#### Finding B-7: Reward catalog hoodie name (line 33644)

{ id: 'hoodie',  name: 'Manna Maids Hoodie',  cost: 200, ... },

Inside the previously-RETIRED `/* legacy reward catalog */` block. Already commented out — NOT executing in production. Listed here only because v11.0.1's earlier audit flagged it. Since the live reward catalog is in the per-business `reward_catalog` Supabase table (Migration 027), this is dead code.

**Recommended action:** delete the entire retired block on next cleanup pass, not urgent.

### 🟡 MEDIUM — needs design but not actively leaking

#### Finding B-8: `RC_CLIENT_ID` (line 14945)

```js
const RC_CLIENT_ID = '9j4d783nueRarrIX5WSYX4';
```

This is the **platform-level** RingCentral OAuth app ID (used for the in-browser OAuth flow). Same for every tenant. Each tenant brings their own RC account + refresh token (Item B from v11.0.2), but the OAuth app that brokers the connection is shared at the Penta platform level.

This is **arguably correct** — analogous to a Stripe publishable key. Not a leak per se. But worth flagging because:
- If you ever spin up a second OAuth app for staging vs prod, this needs to become env-derived
- Should be reviewed alongside the broader Geotab credential migration to make sure the platform-vs-tenant boundary is consistent across all third-party integrations

**Recommended action:** Document explicitly in `docs/06-technical-architecture.md` that this is platform-level config, not tenant. No fix needed unless multi-environment support is needed.

#### Finding B-9: `RC_REDIRECT_URI` (line 14946)

```js
const RC_REDIRECT_URI = 'https://captainclean115-oss.github.io/cleaningapp';
```

Same category — platform-level deploy config. Hardcoded to the prod GitHub Pages URL. Would break if/when Penta moves to a custom domain or staging environment.

**Recommended action:** Move to a build-time / env-var pattern when introducing a staging environment.

### 🟢 LOW — operational config, not tenant-specific

#### Finding B-10: `HOLIDAY_LIBRARY` (line 15503)

Locale-keyed (us / mx / br / cv) array of holidays + greeting bodies. Used for SMS reminders / birthday messages. **Not tenant-specific** — same US holidays apply to every US-based tenant. Safe as-is.

**However:** the greeting bodies (`"Happy Thanksgiving — grateful for you."`) might want to become per-tenant customizable later so operators can match brand voice. Not a multi-tenant blocker. Note as a future feature, not a leak.

#### Finding B-11: `JOB_ISSUE_OPTIONS` (line 14466)

Generic operational dropdown for the employee Report Issue flow: locked out, forgot key, no one home, etc. Generic enough to apply to any cleaning tenant. The `clientSms` templates use "your cleaning team" generically rather than a specific business name. Safe.

#### Finding B-12: `RATE_CATEGORIES` (line 33698)

Generic weekly-rating dimensions (Teammates, Workload, House Difficulty, Equipment & Supplies, Drive Time) in 4 languages. Not tenant-specific.

#### Finding B-13: Third-party API endpoints (multiple)

- `https://api.anthropic.com/v1/messages` — Claire LLM calls (8 occurrences). Platform-level, correct as-is. Already flagged in v11.0.x architecture doc as needing Edge Function migration for credential hygiene.
- `https://platform.ringcentral.com/restapi/...` — RingCentral SMS + OAuth. Platform-level, correct.
- `https://nominatim.openstreetmap.org/...` — geocoding. Platform-level, correct.

All correctly platform-level. No tenant business identifier embedded in any of these URLs.

#### Finding B-14: Demo / placeholder branding

Searched for `demo`, `test_`, `south shore`, similar. Only hits are intentional UI flags marking demo features (e.g., the payments hub modal labeled "Demo — connect processor later") and inline class names (`inv-phase-dot.demo`). No leaks.

### 🟢 NOT-FOUND — categories that came up clean

The following categories were scanned and **no findings** to report:

- **Tax configurations** — no `TAX_RATE`, `SALES_TAX`, `WITHHOLDING` consts. Tax handling not yet implemented in the app.
- **Recurring schedule defaults** — no per-tenant defaults baked in (schedule auto-generation reads from per-client `freq` field, which is per-row in DB).
- **Pricing rules / commission tables** — no `PRICING_TIERS`, `HOURLY_RATE`, `COMMISSION_PCT` consts. The `_qtDiscMap` (quote discount-by-frequency) is product config, not tenant.
- **Image URLs with tenant identity** — searched for `mannamaids.com`, `captainclean`, etc. in src= / url() / `background-image` patterns. No hits (besides the platform `captainclean115-oss.github.io` redirect URI in Finding B-9).
- **Webhook URLs with business IDs** — searched for `/webhook`, `webhook_url`, `.webhook`. No hits.
- **Email templates with tenant signatures beyond what's in B-5/B-6** — checked the entire customer-facing template surface (`tmpl.value`, mailto: builders, quote body). All other templates either already use `PentaTenant.name()` (Item A) or are listed above.
- **Notification configurations** — no per-tenant `NOTIFY_DEFAULTS`, `ALERT_RULES` hardcodes. Real notification config (when it lands) will go through `notification_preferences` table which already exists with `business_id`.
- **Other seed functions like `seedStaffChatsIfEmpty`** — only one other (`_seedDemoRewardPoints` at line 31311) and it's already wrapped in `/* ... */` retirement block. Not active.
